## 1. 플랫폼별 UI 패턴 분석

### 1.1 MediaWiki (Wikipedia)

Wikipedia의 최신 스킨인 **Vector 2022**를 기준으로 분석한다.

#### 네비게이션 구조

| 요소 | 설명 |
|------|------|
| **좌측 사이드바** | 접을 수 있는(collapsible) 사이드바. 메인 네비게이션 링크, 도구 모음 포함 |
| **목차(TOC)** | Vector 2022에서 본문 상단이 아닌 **좌측 사이드바 영역에 고정(sticky)** 배치. 스크롤 시 현재 위치 하이라이트 |
| **브레드크럼** | 카테고리 기반 경로 표시 (예: 카테고리:건축자재 > 몰탈 > 보수몰탈) |
| **페이지 도구** | Vector 2022에서 **우측 상단**으로 이동 (편집, 이력, 토론 등) |
| **상단 고정 헤더** | 스크롤 시에도 검색바와 사용자 메뉴가 상단에 고정 |

#### 문서/페이지 레이아웃

- **제한된 본문 너비**: 가독성을 위해 본문 영역의 최대 너비를 제한 (약 960px)
- **문서 구조**: 제목 > 서론(lead section) > 목차 > 본문(섹션별 H2/H3) > 참고문헌 > 관련 항목 > 카테고리
- **인포박스(Infobox)**: 문서 우측 상단에 핵심 정보를 요약하는 **구조화된 카드**. 제품 위키에서 물성표로 활용 가능
- **각주/참조**: 본문 내 인라인 참조 번호 + 하단 참고문헌 섹션

#### 검색 UI

- **상단 중앙** 검색바 (Vector 2022에서 sticky 헤더에 포함)
- 입력 즉시 **자동완성 드롭다운** (문서 제목 + 본문 발췌 미리보기)
- 검색 결과 페이지: 제목 매칭 우선 표시, 본문 내용 매칭 이후 표시
- 고급 검색: 네임스페이스별 필터링

#### 카테고리/태깅

- **카테고리 시스템**: 문서 하단에 카테고리 링크 표시, 계층적 카테고리 트리 지원
- 카테고리 페이지: 해당 카테고리에 속한 문서 목록 + 하위 카테고리 표시
- 복수 카테고리 지정 가능 (예: `[[카테고리:보수몰탈]]` `[[카테고리:KS F 4042]]`)

#### 첨부파일/문서 처리

- **파일 네임스페이스**: 파일(이미지, PDF 등)을 별도 네임스페이스로 관리
- 파일 페이지에서 메타데이터, 사용처(어떤 문서에서 참조하는지) 확인 가능
- PDF는 직접 뷰어 제공보다는 다운로드 링크 제공이 기본

#### 모바일 대응

- **별도 모바일 스킨(MinervaNeue)** 사용
- Citizen 스킨: 반응형 디자인으로 데스크톱/모바일 단일 코드베이스 지원
- 모바일에서 사이드바는 햄버거 메뉴로 접힘, TOC는 상단 드롭다운으로 변환

#### 몰탈 위키 적용 시 장점

- 인포박스 패턴이 **제품 물성 요약 카드**로 매우 적합
- 카테고리 시스템이 제품 분류(보수몰탈/방수몰탈/바닥몰탈 등)와 자연스럽게 매칭
- 내부 링크([[ ]])를 통한 문서 간 연결이 관련 제품/규격 연결에 효과적

### 1.2 Docusaurus

Meta(Facebook)가 개발한 오픈소스 문서 사이트 생성기.

#### 네비게이션 구조

| 요소 | 설명 |
|------|------|
| **좌측 사이드바** | 계층적 트리 구조. 카테고리 자동 접기 옵션(`autoCollapseCategories`) 지원 |
| **우측 TOC** | 현재 문서의 목차를 우측에 별도로 표시. 스크롤 추적(active state) |
| **상단 네비게이션 바** | 고정(sticky). 프로젝트 로고, 주요 섹션 링크, 검색, 버전 드롭다운, 다크모드 토글 |
| **브레드크럼** | 사이드바 계층 기반 자동 생성 |
| **이전/다음 네비게이션** | 문서 하단에 이전/다음 문서 링크 |

#### 문서/페이지 레이아웃

- **3단 레이아웃**: 좌측 사이드바 | 중앙 본문 | 우측 TOC
- 프론트매터(Frontmatter)를 통한 메타데이터 관리 (title, description, tags, sidebar_position 등)
- MDX 지원: Markdown 내에서 React 컴포넌트 삽입 가능 (인터랙티브 테이블, 탭 등)
- **Admonition 블록**: note, tip, info, caution, danger 등 강조 박스

#### 검색 UI

- **Algolia DocSearch** 통합: Cmd+K / Ctrl+K 단축키로 검색 모달 열기
- 검색 결과에 문서 제목, 섹션 헤딩, 본문 하이라이트 표시
- 로컬 검색 플러그인도 사용 가능

#### 카테고리/태깅

- 사이드바 카테고리로 문서 그룹화 (디렉토리 기반 자동 생성 가능)
- 태그 시스템: 프론트매터에 태그 지정 → 태그 페이지 자동 생성
- 태그 클라우드/목록 페이지 지원

#### 첨부파일/문서 처리

- 정적 파일은 `static/` 폴더에 배치
- Markdown 내 링크로 PDF 등 파일 참조
- 별도 파일 관리 UI는 없음 (Git 기반 관리)

#### 모바일 대응

- 기본 테마가 **완전 반응형**
- 모바일에서 좌측 사이드바는 슬라이드 메뉴, 우측 TOC는 상단 접기식으로 변환
- 하단 네비게이션(이전/다음)은 모바일에서도 유지

#### 몰탈 위키 적용 시 장점

- **버전 관리 시스템**이 제품 규격 변경 이력 관리에 활용 가능
- MDX를 통한 인터랙티브 물성 비교 테이블 구현 가능
- 3단 레이아웃이 기술 문서에 최적화

### 1.3 GitBook

클린한 디자인의 문서 플랫폼. SaaS 기반.

#### 네비게이션 구조

| 요소 | 설명 |
|------|------|
| **좌측 사이드바** | 전체 문서 계층 구조 표시. 접기/펼치기 가능. 검색바, 알림, 스니펫 등 통합 |
| **우측 "On this page"** | 현재 페이지 내 섹션 목차. 자동 생성 |
| **상단 헤더** | 검색바 위치 선택 가능 (중앙 prominent / 우측 상단 subtle) |
| **탭 네비게이션** | 스페이스 간 전환을 위한 상단 탭 |

#### 문서/페이지 레이아웃

- **깔끔하고 넓은 본문 영역**: 충분한 여백(white space) 확보
- 리치 텍스트 편집기: 헤딩, 콜아웃, 이미지, 코드 블록, 테이블 등
- 드래그 앤 드롭으로 콘텐츠 재배열
- **콘텐츠 블록**: 확장 가능한 섹션(expandable), 탭, 힌트 박스 등

#### 검색 UI

- **AI 기반 검색**: 자연어 질문으로 검색 가능 (GitBook AI)
- 검색바 위치를 헤더 중앙(prominent) 또는 우측(subtle) 중 선택 가능
- 헤더를 숨기면 사이드바 내 검색으로 자동 이동
- 스페이스/콘텐츠 전체를 대상으로 통합 검색

#### 카테고리/태깅

- 사이드바의 **그룹(Group)** 기능으로 문서 분류
- 스페이스(Space) 단위로 대분류 관리
- 별도 태그 시스템은 제한적

#### 첨부파일/문서 처리

- 페이지 내 파일 직접 삽입 (이미지, PDF 등)
- Git 저장소와 양방향 동기화 지원
- 파일 버전 관리는 Git을 통해 수행

#### 모바일 대응

- 완전 반응형 디자인
- 모바일에서 사이드바는 오버레이 메뉴로 전환
- 최소한의 UI 요소로 콘텐츠에 집중하는 모바일 경험

#### 몰탈 위키 적용 시 장점

- **AI 검색**이 기술 용어 검색에 유리 (예: "압축강도 40MPa 이상 보수몰탈" 자연어 검색)
- 깔끔한 디자인이 기술 데이터 가독성에 기여
- 콘텐츠 블록(탭, 확장섹션)이 제품별 TDS/MSDS 정리에 활용 가능

### 1.4 Notion

데이터베이스 기반 위키/문서 플랫폼.

#### 네비게이션 구조

| 요소 | 설명 |
|------|------|
| **좌측 사이드바** | 페이지 트리 구조. 즐겨찾기, 워크스페이스 섹션, 개인 섹션 구분 |
| **브레드크럼** | 상단에 페이지 경로 표시 (부모 > 자식 > 현재) |
| **내부 TOC** | 페이지 내 H1/H2/H3 기반 목차 (별도 블록으로 삽입하거나 우측에 자동 표시) |
| **백링크** | 현재 페이지를 참조하는 다른 페이지 목록 표시 |

#### 문서/페이지 레이아웃

- **블록 기반 편집**: 텍스트, 헤딩, 테이블, 데이터베이스, 토글, 콜아웃 등 모든 것이 블록
- **데이터베이스 뷰**: 동일 데이터를 테이블/보드/갤러리/캘린더/타임라인으로 표시
- **위키 전용 기능**:
  - **소유자(Owner)** 속성: 각 페이지의 담당자 지정
  - **검증(Verification)** 속성: 페이지 정확성 확인 날짜 + 만료 알림
  - **Home 뷰**: 자유 배치 가능한 위키 메인 페이지
  - **All pages 뷰**: 전체 페이지를 테이블로 조회
- 페이지 속성이 **수평(horizontal)으로 표시**되어 공간 효율적

#### 검색 UI

- **Cmd+K / Ctrl+K** 전역 검색
- 최근 방문 페이지 표시
- 필터: 생성자, 날짜, 워크스페이스 등
- AI 기반 검색 (Notion AI Q&A)

#### 카테고리/태깅

- 데이터베이스 **속성(Property)**으로 다중 분류 가능:
  - Select / Multi-select 속성으로 카테고리/태그 구현
  - 필터, 정렬, 그룹화로 다양한 뷰 생성
- 위키 뷰에서 필터링: 검증 상태별, 카테고리별, 소유자별

#### 첨부파일/문서 처리

- 페이지 내 파일 블록으로 직접 업로드 및 미리보기
- PDF 미리보기 내장
- 이미지, 동영상, 오디오 등 다양한 파일 타입 지원
- 파일 크기 제한 있음 (플랜에 따라 다름)

#### 모바일 대응

- 네이티브 모바일 앱 (iOS, Android) 제공
- 반응형 웹도 지원
- 모바일에서도 데이터베이스 뷰 전환 가능

#### 몰탈 위키 적용 시 장점

- **데이터베이스 뷰**가 제품 목록(테이블), 카테고리별 분류(보드), 일정 관리(캘린더)에 매우 적합
- **검증(Verification) 시스템**이 기술 문서의 정확성 관리에 적합 (예: 시험성적서 유효기간)
- **속성 기반 필터링**이 제품 물성 비교에 활용 가능

### 1.5 Confluence

Atlassian의 엔터프라이즈 위키 플랫폼.

#### 네비게이션 구조

| 요소 | 설명 |
|------|------|
| **좌측 사이드바** | 2025 리디자인으로 **모든 네비게이션이 사이드바로 통합**. 페이지 트리, 스페이스 단축키, 블로그 링크 포함 |
| **페이지 트리** | 부모-자식 관계의 계층적 페이지 구조. 사이드바에서 현재 위치 근처 200페이지 표시 |
| **브레드크럼** | 스페이스 > 부모 페이지 > 현재 페이지 경로 표시 |
| **스페이스 시스템** | 대분류를 스페이스로 분리 (예: "제품DB", "기술문서", "시험성적서" 스페이스) |
| **앱 메뉴** | 상단 네비게이션에 앱 통합 메뉴 (2025 신규) |

#### 문서/페이지 레이아웃

- **매크로 시스템**: 목차, 페이지 트리, 차트, 상태 표시 등 다양한 매크로 삽입
- **페이지 속성(Page Properties)** 매크로: 키-값 형태의 구조화된 데이터 표시
- **레이블(Label)**: 페이지에 레이블 태그 부착
- 템플릿 시스템: 반복 사용하는 페이지 구조를 템플릿으로 저장

#### 검색 UI

- **글로벌 검색바**: 상단에 위치
- CQL(Confluence Query Language)로 고급 검색
- 스페이스, 작성자, 날짜, 레이블 등으로 필터링
- 최근 검색 이력 표시

#### 카테고리/태깅

- **레이블(Label)** 시스템: 자유 형식 태그
- Content by Label 매크로로 특정 레이블의 문서 목록 자동 생성
- 스페이스 + 레이블 조합으로 2차원 분류 가능

#### 첨부파일/문서 처리

- 페이지별 **첨부파일 영역**: 파일 업로드 및 목록 관리
- 파일 버전 관리 지원
- Office 문서 미리보기 내장
- 매크로를 통한 첨부파일 목록 표시

#### 모바일 대응

- 모바일 앱 제공 (iOS, Android)
- 반응형 웹 인터페이스

#### 몰탈 위키 적용 시 장점

- **스페이스 분리**가 "제품DB", "규격문서", "시공사례" 등 대분류에 적합

## 3. 기술 제품 데이터 UI 관련 추가 조사

### 3.1 제품 사양서(Spec Sheet) UI 베스트 프랙티스

Baymard Institute의 이커머스 UX 리서치 결과를 기반으로 한 권장사항:

#### 핵심 문제점 (피해야 할 것)
- **38%의 사이트**가 관련 사양을 소그룹으로 묶지 않아 스캔 가능성(scannability)이 떨어짐
- **32%의 사이트**가 같은 제품 유형 간 사양표 일관성이 없음
- **50%의 사이트**가 일반 사용자가 이해하기 어려운 전문 용어를 설명 없이 사용

#### 권장 UI 패턴

1. **그룹화**: 관련 사양을 소섹션으로 그룹화 (예: "강도 물성" / "내구성" / "시공 조건")
2. **일관된 레이아웃**: 같은 카테고리의 모든 제품이 동일한 사양 항목을 동일한 순서로 표시
3. **시각적 구분**: 그룹 간 구분선, 배경색 차별화, 카드형 컨테이너 활용
4. **점진적 공개(Progressive Disclosure)**: 핵심 사양은 바로 표시, 상세 사양은 확장(expand) 가능하게
5. **반응형 그리드**: 모바일 2열 → 태블릿 3-4열 → 데스크톱 4-5열

### 3.2 건축자재 데이터베이스 UI 레퍼런스

#### Sweets (건축 제품 데이터베이스)
- 제품 정보, CAD 상세도, BIM 객체, 사양서, 카탈로그, 녹색 제품 정보를 통합 제공
- 브랜드별, 기능 카테고리별 검색 가능

#### 주요 UI 패턴
- **패싯 검색(Faceted Search)**: 왼쪽에 필터 패널, 오른쪽에 결과 목록
- **비교 기능**: 복수 제품 선택 후 사양 비교 테이블 표시
- **제품 카드 그리드**: 썸네일 + 핵심 사양 2-3개 + 카테고리 배지

### 3.3 내부 지식 베이스 UI 베스트 프랙티스 (2025)

#### 조직 및 구조
- 내부 조직도가 아닌 **사용자의 멘탈 모델**에 맞춰 콘텐츠를 구성
- 기본에서 심화로 흐르는 논리적 계층 구조 구축
- 핵심 토픽이 나열된 매력적인 홈페이지 설계

#### 검색 및 발견 가능성
- **2025년 사용자는 Google 수준의 검색 속도**를 기대
- 동의어 및 태깅 필수: 기능명이 "압축강도"라면 "compressive strength", "comp. str." 등도 검색 가능해야 함
- 오타 허용(fuzzy search) 지원 필요

#### 콘텐츠 유지보수
- 전담 지식 관리자(Knowledge Manager) 지정
- 정기적 콘텐츠 검토 및 아카이브 프로세스
- 표준화된 포맷, 명명 규칙, 승인 프로세스 수립

### 3.4 문서 네비게이션 설계 원칙

"I'd Rather Be Writing" 블로그의 기술 문서 네비게이션 연구 결과:

| 원칙 | 설명 | 몰탈 위키 적용 |
|------|------|---------------|
| **계층 구조(Hierarchy)** | 부모-자식 구조로 정보 조직. 1-2단계로 제한 | 제품 카테고리 > 제품 > 세부 정보 |
| **점진적 공개(Progressive Disclosure)** | 포탈 홈 → 제품 홈 → 섹션 홈 → 개별 페이지 | 메인 > 보수몰탈 > VAP-R100 > TDS |
| **몰입(Immersion)** | 위키피디아식 인라인 링크로 문맥 내 이동 | 본문에서 관련 규격, 제품 직접 링크 |
| **욕망의 경로(Desire Lines)** | 인기 콘텐츠를 분석해 눈에 띄게 배치 | 자주 조회되는 제품/규격을 홈에 표시 |
| **모듈성(Modularity)** | 자체 완결적 콘텐츠 단위. 800-3,000자 권장 | 하나의 제품 페이지 = 완전한 정보 단위 |
| **길 찾기(Wayfinding)** | 워크플로 맵, TOC, "다음 단계" 제공 | 관련 제품, 대체 제품 링크 |

8. 관련 규격

10. 카테고리/태그

1. 규격 정보 카드 (규격번호, 제정일, 개정일)
2. 개요
3. 주요 요구사항 (테이블)
4. 시험 방법 요약
5. 적합 제품 목록 (자동 생성)
6. 관련 규격
7. 참고문헌

#### 시공사례 페이지 템플릿

## 7. 결론 및 우선순위 권장

### 최우선 구현 (Phase 2 디자인 단계)

1. **3단 레이아웃 시스템**: 사이드바 + 본문 + 우측 패널 (TOC/물성카드)
2. **제품 인포박스 컴포넌트**: 핵심 물성을 한눈에 보여주는 카드
3. **물성 데이터 테이블**: 그룹화 + 합격/불합격 시각화 + 탭 전환
4. **검색 UI (Ctrl+K 모달)**: 제품/규격/문서 통합 검색
5. **카테고리 + 태그 시스템**: 계층형 카테고리 + 평면형 태그 이중 분류

### 차순위 구현

6. 제품 비교 기능 (체크박스 선택 → 비교 테이블)
7. PDF 인라인 뷰어
8. 문서 유효기간 관리 (인증서, 시험성적서)
9. 반응형 모바일 최적화
10. 다크모드

### 핵심 설계 원칙

> **"기술 제품 데이터는 일반 텍스트 위키와 다르다."**

- **구조화된 데이터가 핵심**: 자유 형식 텍스트보다 물성표, 사양 카드, 규격 정보가 더 중요
- **비교 가능성(Comparability)**: 같은 카테고리 제품은 동일한 형식으로 데이터 표시
- **신뢰성 표시**: 시험성적서 날짜, 인증 유효기간 등 데이터의 신뢰성 메타데이터 필수
- **빠른 접근**: Ctrl+K 검색, 인포박스, 사이드바 트리를 통해 3클릭 이내 원하는 정보 도달
